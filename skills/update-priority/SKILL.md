# Update Priority Workflow

## 1. 适用场景

在以下场景触发：

1. 一次 Get-Info 任务成功找到有效来源并完成持久化。
2. 需要根据近期真实查询和高质量命中情况调整站点优先级。
3. 需要记录关键词热度、最近查询时间和命中站点。

不要在以下场景触发：

1. 搜索失败，没有得到有效来源。
2. 页面抓取失败，没有形成有效知识。
3. 只是猜测某站点可能更好，没有证据支撑。

## 2. 职责边界

本 skill 负责：

1. 读取 `data/priority.json`。
2. 读取或更新 `data/keywords.db`。
3. 基于真实任务结果更新关键词计数、最后查询时间、站点优先级参考值。
4. 回填 `priority.json` 的 `official_domains` 白名单（当 get-info-workflow 报告 LLM 发现了新的高置信官方域名时）。

本 skill 不负责：

1. 发起外部搜索。
2. 清洗网页内容。
3. 替 QA Agent 做回答。

## 3. 更新原则

1. 只依据真实任务结果更新，不凭主观印象更新。
2. 命中高质量结果的站点可以适度提升权重。
3. 连续无产出的站点可以在后续人工策略中降低权重，但不要在单次失败中激进降权。
4. 同一个关键词应记录查询次数与最后查询时间。
5. 更新必须可追踪、可解释。

## 4. 执行流程

### 步骤1: 读取当前状态

1. 读取 `data/priority.json`。
2. 读取 `data/keywords.db`。
3. 确认本次任务涉及哪些站点、关键词、主题。

### 步骤2: 汇总本次任务结果

至少汇总以下信息：

1. 使用了哪些查询。
2. 哪些站点返回了有效页面。
3. 哪些页面最终被持久化。
4. 哪些关键词真正帮助命中了高质量内容。

### 步骤3: 更新关键词库

1. 对真实使用且有效的关键词增加查询计数。
2. 更新 `last_query_at`。
3. 保持站点与关键词之间的关联。

### 步骤4: 更新优先级配置

1. 刷新 `last_update`。
2. 对稳定产出高质量内容的站点适度增加优先级。
3. 对新发现的高价值关键词补充进对应站点列表。
4. 不要写入与本次任务无关的主题词。

### 步骤5: 回填 `official_domains` 白名单

当 `get-info-workflow` 的来源分类步骤中 LLM 判定新的高置信度官方域名时，回填到 `priority.json.official_domains`：

1. 只回填 LLM 标记为高置信度的 official 域名（置信度不足的不写入，避免污染白名单）。
2. 幂等写入：已存在的域名直接跳过。
3. 写入格式与匹配规则：
   - **纯域名**：格式统一为小写域名（如 `docs.anthropic.com`），匹配时子域名自动匹配父域名。
   - **域名+路径前缀**：当 URL 的路径部分也具有官方文档特征时（如 `www.coze.com/open/docs`、`github.com/anthropics`），写入完整域名+路径前缀。匹配时 URL 必须以此项为前缀才算命中。
   - 优先写入更精确的模式：如果 `www.coze.com/open/docs` 已在白名单，就不需要再单独写入 `www.coze.com`（路径前缀已覆盖）。
4. 维护和审计：白名单仅作为分类加速通道，不是安全边界；用户可随时手动编辑 `priority.json` 刪除误收项。

## 5. 失败策略

1. JSON 读取失败时直接报错，不要覆盖原文件。
2. SQLite 更新失败时直接报错，不要假装优先级已同步。
3. 任一步骤失败，都要明确指出是 `priority.json` 还是 `keywords.db` 更新失败。
